## 二、OpenClaw LLM 请求头伪装 Claude Code

### 修改方式
编辑 `~/.openclaw/openclaw.json`，在 `models.providers` 对应 provider 中添加 `headers` 字段：

```json
"headers": {
  "User-Agent": "claude-code/2.1.112",
  "X-Client-Name": "claude-code"
}
```

当前已对 default (MiniMax) 和 zhipu (GLM) 两个 provider 都添加了此伪装头。

### 生效方式
```bash
kill -s USR1 $(pgrep -f openclaw-gateway)
```

### 验证方法
```bash
curl -s -X POST "https://open.bigmodel.cn/api/coding/paas/v4/chat/completions" \
  -H "Authorization: Bearer <key>" \
  -H "User-Agent: claude-code/2.1.112" \
  -d '{"model":"glm-5.1","messages":[{"role":"user","content":"hi"}],"stream":false}'
```

### 原理
OpenClaw 使用 OpenAI Node SDK，`model.headers` 通过 SDK 的 `defaultHeaders` 注入，覆盖 SDK 默认的 `User-Agent: OpenAI/JS x.x.x`。`x-stainless-*` 头由 SDK 强制添加，无法移除，但 curl 模拟测试证明这些头不影响 GLM 响应。

### 注意
- SIGUSR1 热加载会让 gateway 完整重启（而非真正热加载），会导致约 1-2 分钟的服务中断
- 改 `models.json` 后不一定需要 SIGUSR1，下次请求时会自动使用新配置

## 三、GLM 白天限流问题（未解决）

### 现象
- OpenClaw 在白天 10:00-20:00 期间使用 GLM (glm-5.1) 会收到 `429 rate limit` 错误
- 同一 key、同一 IP、同一 baseUrl，Hermes 在同样时段却正常
- 直接 curl 模拟任何请求组合都完全正常

### 排查结论
- ✅ User-Agent 伪装无效（改了还是被限）
- ✅ `originator: openclaw` 头无效
- ✅ 路径 `/coding/paas/v4` 不是决定因素（Hermes 改成同样路径后白天仍正常）
- ✅ 直接 curl 测试完全正常
- ❌ 真正原因仍未定位，怀疑与 OpenClaw 内部请求并发/时序特征有关

### 关键线索
- Hermes 是 Python 写的单进程，OpenClaw 是 Node.js（可能存在多并发请求）
- OpenClaw 发请求时 `x-stainless-*` 头会暴露用的是 OpenAI SDK，但 curl 模拟证明这些头不触发限流
- GLM 可能按会话粒度或请求时序特征做限流

### 待办
- 联系 GLM/智谱技术支持查被限原因
- 或在 GLM 控制台查看被限时间段的请求记录

## 五、openclaw.json 修复

### 问题
```json

  "/glm": "/model zhipu/glm-5.1",
  "/minimax": "/model default/MiniMax-M2.7"

```
OpenClaw 不认识 `/glm` 这种以 `/` 开头的 command key，报 Config invalid。

### 修复
移除 commands 中的 `/glm` 和 `/minimax` 快捷命令定义（改用 alias 机制或其他方式）。

## 六、Docker 存储迁移注意

### 软链接方案不可行
把 `/var/lib/docker/overlay2` 整个目录软链接到新位置会失败，原因是 Docker 在内部数据库存储的是镜像层的绝对路径引用，事后软链接无法欺骗这些已记录的路径，导致 OCI runtime 报 `invalid rootfs: not an absolute path, or a symlink`。

### 正确迁移方式
在 `/etc/docker/daemon.json` 中加 `"data-root": "/home/data/docker"`，然后 mv 整个 docker 目录，再重启 dockerd。
