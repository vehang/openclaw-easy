# 产品设计规范（PRODUCT-DESIGN.md）

> 本文件定义产品设计规范，确保设计专业、有个性、去 AI 化。

---

## 一、设计原则

### 核心原则

```
专业 > 模板    有个性 > 千篇一律    用户体验 > 视觉炫技
```

### 设计哲学

- **每个像素都有意义** - 不堆砌元素，每个设计决策都有理由
- **差异化设计** - 避免与市面模板雷同，做出识别度
- **用户中心** - 设计服务于用户目标，不是设计师的自我表达

---

## 二、设计禁止项

### ❌ 配色禁止

| 禁止项 | 原因 |
|--------|------|
| 紫色/靛蓝色渐变 (#6366F1, #8B5CF6) | AI 生成标配色 |
| 纯平背景色 | 缺乏层次感 |
| Tailwind 默认色板 | 太常见，缺乏个性 |
| 高饱和度霓虹色 | 廉价感 |
| 蓝紫渐变 + 白色文字 | SaaS 模板标配 |

### ❌ 布局禁止

| 禁止项 | 原因 |
|--------|------|
| Hero + 三卡片布局 | AI 生成标配 |
| 完美居中对齐 | 缺乏节奏感 |
| 等宽多栏 | 呆板无趣 |
| 12 列栅格死板应用 | 缺乏变化 |
| 每个区块都有 CTA 按钮 | 过度营销 |

### ❌ 组件禁止

| 禁止项 | 替代方案 |
|--------|----------|
| Shadcn/Material UI 默认样式 | 深度定制或自建 |
| Emoji 作为功能图标 | 使用 Iconify 图标库 |
| 线性动画 (ease-in-out) | 加入缓动变化 |
| 圆角全统一 | 根据层级差异化 |

### ❌ 文案禁止

| 禁止项 | 说明 |
|--------|------|
| 高深专业名词 | 用户看不懂 |
| Lorem Ipsum | 必须用真实文案 |
| 被动语态和长句 | 每 15 字一句 |
| "打造极致体验" | 空洞无意义 |

## 三、设计必须项

### ✅ 配色要求

| 要求 | 说明 |
|------|------|
| 背景有纹理 | 噪点、渐变、图案至少一种 |
| 主色 < 3 种 | 一个品牌色 + 1-2 辅助色 |
| 有温度的灰 | 不用纯灰 (#808080)，用有色相的灰 |
| 对比度 > 4.5:1 | 确保可读性 |
| 暗色模式适配 | 必须提供暗色方案 |

### ✅ 布局要求

| 要求 | 说明 |
|------|------|
| 不对称布局 | 至少一处打破对称 |
| 视觉重心明确 | 用户第一眼看哪里是设计好的 |
| 呼吸感 | 留白至少占 30% |
| 层级分明 | 通过大小、颜色、间距区分层级 |
| 响应式 | 移动端优先，适配 375px - 1920px |

### ✅ 图片系统

| 用途 | 来源 |
|------|------|
| 图标 | [Iconify](https://iconify.design) |
| 占位图 | [Picsum Photos](https://picsum.photos) |
| 真实图片 | [Pexels](https://www.pexels.com) |
| 插画 | [unDraw](https://undraw.co) |

### ✅ 文案要求

| 要求 | 示例 |
|------|------|
| 口语化 | "开始使用" 不是 "立即开始您的旅程" |
| 具体化 | "3 分钟完成注册" 不是 "快速注册" |
| 短句为主 | 每句不超过 15 个字 |
| 有数字 | 用户喜欢具体数字 |
| 场景代入 | "当你想...时" |

## 四、设计工作流

### 设计阶段

```
需求理解 → 信息架构 → 原型设计 → 视觉设计 → 交付验收
```

### 各阶段输出

| 阶段 | 输出物 | 质量标准 |
|------|--------|----------|
| 需求理解 | 用户故事、场景描述 | 用户原话，场景具体 |
| 信息架构 | 页面流程图、内容结构 | 层级清晰，流程顺畅 |
| 原型设计 | 线框图、交互说明 | 核心流程可走通 |
| 视觉设计 | 高保真设计稿 | 符合设计规范 |
| 交付验收 | 设计规范文档、切图 | 开发可落地 |

### 设计审查

每个设计完成后，必须进行以下审查：

#### 审查清单

```
□ 配色是否避免了 AI 标配色？
□ 布局是否避免了模板化结构？
□ 组件是否深度定制？
□ 文案是否说人话？
□ 是否有不对称/个性化元素？
□ 暗色模式是否适配？
□ 响应式是否考虑？
□ 对比度是否达标？
□ 是否有留白和呼吸感？
□ 整体是否有识别度？
```

## 五、技术选型指南

### 根据项目类型选择

| 项目类型 | 推荐技术栈 | 说明 |
|----------|------------|------|
| 营销落地页 | Tailwind + 静态 HTML | 快速开发，加载快 |
| SaaS 后台 | React + Tailwind + Shadcn | 组件丰富，开发效率高 |
| 内容网站 | Next.js + Tailwind | SEO 友好 |
| 移动优先 | PWA + Tailwind | 跨平台 |

### 框架选择原则

1. **优先选择熟悉的** - 效率 > 新技术
2. **考虑维护成本** - 团队能否长期维护
3. **社区活跃度** - 遇到问题能找到答案
4. **性能要求** - 根据业务场景选择

### 各阶段 Skills 使用详解

#### 阶段 1: 需求理解

**目标**: 理解用户真实需求

**输出**:
- 用户故事
- 场景描述
- 验收标准

**规范**:
- 记录用户原话，不要自己转述
- 场景具体化：谁、在什么场景、遇到什么问题
- 验收标准可量化

---

#### 阶段 2: 信息架构

**目标**: 组织内容结构

**输出**:
- 页面流程图
- 内容结构图
- 导航设计

**规范**:
- 层级不超过 3 层
- 用户 3 步内完成核心操作
- 导航标签不超过 7 个

---

#### 阶段 3: 原型设计

**使用 Skill**: `ui-ux-design`

**触发方式**:
```
"使用 ui-ux-design 指导设计用户登录流程"
```

**Skill 能力**:
- 移动端优先设计
- WCAG 2.2 可访问性
- Tailwind + Shadcn 集成
- 微交互设计

**输出**:
- 线框图
- 交互说明
- 状态流转图

---

#### 阶段 4: 视觉设计

**使用 Skills**: `frontend-design-3`, `tailwind-design-system`

**触发方式**:
```
"使用 frontend-design-3 设计一个去AI化的登录页面"
"使用 tailwind-design-system 构建组件库"
```

**frontend-design-3 能力**:
- 避免 AI 美学
- 专业级前端界面
- 有个性的设计

**tailwind-design-system 能力**:
- CVA 变体系统
- 设计令牌
- 暗色模式
- 响应式网格

**去 AI 化检查**:
```markdown
□ 无紫色/靛蓝渐变
□ 无 Hero + 三卡片布局
□ 组件深度定制
□ 有不对称元素
□ 有留白和呼吸感
```

---

#### 阶段 5: 文案优化

**使用 Skill**: `humanizer`

**触发方式**:
```
"使用 humanizer 优化这段文案"
```

**Skill 能力**:
- 检测 AI 写作痕迹
- 修复空洞词汇
- 去除模式化表达
- 使文案更自然

**检查清单**:
```markdown
□ 无"极致体验"等空洞词
□ 无被动语态和长句
□ 每句不超过 15 字
□ 有数字和场景
```

---

#### 阶段 6: 设计审查

**使用 Skill**: `ui-ux-design`

**审查命令**:
```
"对当前设计进行 UI/UX 审查"
```

**审查维度**:
- 配色审查
- 布局审查
- 可访问性审查
- 响应式审查

---

### Skills 协同示例

**完整设计流程示例**:

```
用户: "帮我设计一个用户登录页面"

1. 需求理解
   - 用户故事: 用户需要快速登录
   - 场景: 移动端，输入手机号验证码登录
   - 验收: 3 步内完成登录

2. 使用 ui-ux-design 设计原型
   - 移动端优先
   - 最小化输入
   - 清晰的反馈

3. 使用 frontend-design-3 视觉设计
   - 避免紫色渐变
   - 不对称布局
   - 有温度的配色

4. 使用 humanizer 优化文案
   - "输入手机号" 不是 "请输入您的手机号码"
   - "获取验证码" 不是 "获取短信验证码"

5. 审查确认
   - 配色: 无 AI 标配色 ✓
   - 布局: 不对称 ✓
   - 文案: 说人话 ✓
```

### 验收标准

| 标准 | 说明 |
|------|------|
| 可实现 | 开发能落地，不是空中楼阁 |
| 可维护 | 设计系统清晰，易于扩展 |
| 可测量 | 有明确的验收指标 |
| 可复用 | 组件可跨页面复用 |

## 八、去 AI 化检查

### 设计完成后自检

```markdown
## AI 痕迹检查

- [ ] 配色是否有紫色/靛蓝渐变？
- [ ] 布局是否是 Hero + 三卡片？
- [ ] 组件是否用了默认样式？
- [ ] 动画是否都是线性？
- [ ] 文案是否有"极致体验"等空洞词？
- [ ] 是否完美居中对齐？
- [ ] 整体是否像某个模板？

→ 如果有 3 个以上，需要重新设计
```
