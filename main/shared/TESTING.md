## 二、各阶段详解

### 阶段 1: 需求对照

**目标**: 确保测试覆盖所有需求

**工作内容**:
1. 读取 PRD 文档
2. 提取验收标准
3. 确定测试范围
4. 划分优先级

**输出**:
```markdown
## 测试范围

### P0 (必测)
- 核心功能流程

### P1 (重要)
- 主要功能分支

### P2 (一般)
- 边界情况
```

---

### 阶段 2: 测试设计

**使用 Skills**: `test-master`, `testing-patterns`

**test-master 能力**:
- 测试策略设计
- 自动化框架搭建
- 覆盖率分析
- 性能测试

**testing-patterns 能力**:
- 单元测试模式
- 集成测试模式
- E2E 测试模式
- 多语言支持 (Jest/Vitest/pytest/Go/Rust)

**测试用例格式**:
```markdown
## TC-001: 用户登录成功

**优先级**: P0
**类型**: 功能测试
**前置条件**: 用户已注册

**测试步骤**:
| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 输入正确账号 | 显示账号 |
| 2 | 输入正确密码 | 显示密码掩码 |
| 3 | 点击登录 | 登录成功 |

**验证点**:
- [ ] 登录成功提示
- [ ] 跳转首页
- [ ] 用户信息正确
```

---

### 阶段 3: 测试执行

**使用 Skill**: `test-runner`

**能力**:
- 单元测试
- 集成测试
- E2E 测试
- 多语言支持 (TypeScript/Python/Swift)

**测试命名规范**:
```javascript
// 格式: should_xxx_when_xxx
describe('用户登录', () => {
  test('should_login_success_when_credentials_correct', () => {
    // ...
  });
  
  test('should_show_error_when_password_wrong', () => {
    // ...
  });
});
```

**执行命令**:
```bash
# 运行所有测试
npm test

# 运行单个文件
npm test -- login.test.js

# 覆盖率报告
npm test -- --coverage
```

---

### 阶段 4: API 测试

**使用 Skill**: `api-tester`

**能力**:
- HTTP 请求测试 (GET/POST/PUT/DELETE)
- 自定义 Headers
- JSON Body 支持
- 健康检查

**API 测试格式**:
```markdown
## API-001: 登录接口

**接口**: POST /api/v1/auth/login
**优先级**: P0

**请求**:
```json
{
  "account": "test@example.com",
  "password": "test123"
}
```

**预期响应**:
```json
{
  "code": 0,
  "data": { "token": "..." }
}
```

**验证点**:
- [ ] 状态码 200
- [ ] 返回 token
- [ ] 响应时间 < 1s
```

---

### 阶段 5: 问题记录

**目标**: 记录问题并转交开发

**Bug 报告格式**:
```markdown
## BUG-001: 登录超时未提示

**严重程度**: 中
**发现时间**: 2026-03-16
**发现人**: 测试团队

**问题描述**:
网络超时时，未显示友好提示

**重现步骤**:
1. 输入账号密码
2. 断开网络
3. 点击登录

**预期结果**: 显示"网络异常，请重试"
**实际结果**: 页面无响应

**转交**: 开发团队
**状态**: 待修复
```

---

### 阶段 6: 回归测试

**触发条件**:
- Bug 修复后
- 功能优化后
- 发布前

**测试范围**:
1. 修复功能的验证测试
2. 相关功能的回归测试
3. 核心流程的冒烟测试

---

### 阶段 7: 发布检查

**使用 Skill**: `test-publish-check`

**能力**:
- 代码质量检查
- API 测试
- 部署检查清单
- 版本管理
- 上线前审查
- 回归测试

**发布检查清单**:
```markdown
## 发布检查

### 代码质量
- [ ] 代码审查通过
- [ ] 无明显性能问题
- [ ] 无安全漏洞

### 测试
- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] 回归测试通过
- [ ] 覆盖率达标 (>80%)

### 部署
- [ ] 环境配置正确
- [ ] 数据库迁移完成
- [ ] 依赖服务正常

### 文档
- [ ] 更新日志已写
- [ ] API 文档已更新
```

## 三、测试 Skills 使用指南

### 已安装的测试 Skills

| Skill | 用途 | 使用场景 |
|-------|------|----------|
| **test-runner** | 测试执行 | 运行单元/集成/E2E 测试 |
| **test-master** | 测试策略 | 测试设计、覆盖率分析 |
| **testing-patterns** | 测试模式 | 单元/集成/E2E 测试模式 |
| **test-publish-check** | 发布检查 | 上线前质量检查 |
| **api-tester** | API 测试 | HTTP 接口测试 |

### 使用方式

**设计测试策略**:
```
使用 test-master 分析测试范围和策略
```

**运行测试**:
```
使用 test-runner 执行测试用例
```

**API 测试**:
```
使用 api-tester 测试接口
```

**发布前检查**:
```
使用 test-publish-check 进行上线前审查
```

## 四、测试类型

### 单元测试

| 项目 | 标准 |
|------|------|
| 覆盖率 | > 80% |
| 命名 | should_xxx_when_xxx |
| 结构 | Arrange-Act-Assert |
| 速度 | < 1s/用例 |

### 集成测试

| 项目 | 标准 |
|------|------|
| 覆盖 | 关键流程 |
| 数据 | 使用测试数据 |
| 环境 | 独立测试环境 |
| 清理 | 测试后数据清理 |

### E2E 测试

| 项目 | 标准 |
|------|------|
| 覆盖 | 核心用户场景 |
| 稳定性 | 无 Flaky 测试 |
| 维护 | 及时更新用例 |

## 六、质量标准

### 测试通过标准

| 类型 | 标准 |
|------|------|
| 单元测试 | 覆盖率 > 80%，全部通过 |
| 集成测试 | 关键流程通过 |
| E2E 测试 | 核心场景通过 |
| API 测试 | 所有接口正常 |

```markdown

□ 无未关闭的严重 Bug
□ 覆盖率达标
□ 发布检查通过
```
