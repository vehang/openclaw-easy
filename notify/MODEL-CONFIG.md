## 二、竞争机制

### 适用场景

**关键任务**启用竞争模式：
- ✅ 产品设计（影响全局）
- ✅ 架构设计（影响全局）

**普通任务**使用单模型：
- 开发编码
- 测试执行
- 文档编写

---

### 竞争流程

```
┌─────────────────────────────────────────────────────────────┐
│                    竞争模式流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Step 1: 并行生成                                           │
│  ┌──────────────┐          ┌──────────────┐                │
│  │  主模型       │          │  竞争模型     │                │
│  │  qwen3.5-plus│          │  kimi-k2.5   │                │
│  │              │          │              │                │
│  │   方案 A     │          │   方案 B     │                │
│  └──────────────┘          └──────────────┘                │
│                                                             │
│  Step 2: 评判选择                                           │
│           ┌──────────────────┐                              │
│           │   评判模型        │                              │
│           │   qwen3-max      │                              │
│           └──────────────────┘                              │
│                     │                                       │
│                     ▼                                       │
│  Step 3: 输出最优方案                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

---

### 评判标准

评判模型从以下维度评分（1-10分）：

| 维度 | 说明 | 权重 |
|------|------|------|
| **完整性** | 功能是否覆盖全面 | 25% |
| **合理性** | 设计是否符合逻辑 | 25% |
| **可实现性** | 技术上是否可行 | 20% |
| **创新性** | 是否有亮点 | 15% |
| **清晰度** | 表达是否清晰 | 15% |

**总分 = 各维度分数 × 权重之和**

## 三、实现方式

### 产品设计阶段

```yaml
竞争配置:
  主模型: kimi-k2.5
  竞争模型: qwen3-coder-plus
  评判模型: glm-5

执行步骤:
  1. 并行调用主模型和竞争模型，生成两个设计方案
  2. 用评判模型对比两个方案，评分并选择最优
  3. 输出最优方案作为最终产出
```

### 技术设计阶段

```yaml
单模型配置:
  模型: kimi-k2.5
  
原因:
  - 输出最详细
  - 有完整的错误码设计
  - 表格格式清晰
```

### 开发阶段

```yaml
单模型配置:
  模型: glm-5
  
原因:
  - 代码能力够用
  - 简洁清晰
  - 避免代码风格不一致
  
执行方式:
  - 使用 tmux + Claude Code CLI
  - 实时查看进度，可手动干预
```

### 测试阶段

```yaml
单模型配置:
  模型: glm-5
  
原因:
  - 逻辑分析强
  - Bug 定位准确
  - 效率优先
```

## 五、调用示例

### 竞争模式调用（产品设计）

```javascript
// 产品设计 - 竞争模式
const result = await runCompetition({
  task: "设计用户登录功能的产品方案",
  models: ["kimi-k2.5", "qwen3-coder-plus"],
  judge: "glm-5",
  criteria: ["完整性", "合理性", "可实现性", "创新性", "清晰度"]
});
```

### 单模型调用（技术设计）

```javascript
// 技术设计 - 单模型
const result = await runSingle({
  task: "设计用户登录功能的API接口",
  model: "kimi-k2.5"
});
```

### 开发阶段（tmux + Claude Code）

```bash
# 启动 tmux 窗口运行 Claude Code
tmux new-session -d -s coding "claude --model glm-5"
tmux attach -t coding

# 或使用 OpenClaw spawn
sessions_spawn({
  model: "glm-5",
  task: "实现用户登录功能代码"
})
```
